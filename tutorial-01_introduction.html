01. Introduction

What is an Application programming interface (API)?

Here, it is a way of providing controls for, and access to, a database connected to the Internet. It is more properly called a Web API - there are others types of API.

API standards

Just to be clear, this API doesn't follow any existing API standards. That's a decision that wasn't taken lightly, and the original intention was to adhere to a standard if possible.

Two API standards in particular were examined:

  • OpenAPI
  • SDMX (Statistical Data and Metadata eXchange)

These, however, address use-cases that diverge significantly from the needs of this API. The use-case for this API is to allow customisable access to data from statistical releases. It is intended to be used by code, with the data being provided in a format suitable for further processing. The data is stored in a database rather than spreadsheats, which means the data can be reshaped, sorted and filtered in any way the user needs.

How does this API differ from OpenAPI and SDMX, and why?

In line with user-centred design principles, the intention is to allow users to select the features they want from the dataset, in any order, from any of the data-dimensions (these are explained in 05_adding_data).

There are some prescriptive elements at the start of the API query necessary to allow the API to narrow down the user's choice of dataset in order to get the correct dimensions.

However, once the dataset has been established, the remainder of the API call is intended to be unstructured.

This is a consequence of allowing users to select any number of features from a list that can be in the hundreds. A rigid hierarchical API call structure, as in OpenAPI or SDMX, would quicky become a significant cognitive burden on users.

In more detail, the idea is to provide access to a dataset such that:

  • the request is sufficiently expressive i.e. the user should be able to specify their requirements with sufficient precision to identify a single datum within the dataset, through to subsets, or the entire dataset
  • it is accesible via the Internet
  • there is a clear methodology for the data owner to make their data available
  • the user can choose the format of the data returned
  • the user can choose the shape of the data returned
  • it is self-describing and discoverable (only partially implemented in this version of the API)

Other documentation: the API user guide

The API allows clients (particularly code, but anything like a browser or tools like curl), to query it and get data. How to do this is explained in the manual that accompanies each dataset in the API, as it requires contextual information about the dataset. As an example, see the documentation for the DUKES dataset.

To access the API and get data from it, the client/user needs to

  • ascertain what services are available
  • identify how to format service requests
  • know how to send and receive requests and responses
  • understand the format of the response
